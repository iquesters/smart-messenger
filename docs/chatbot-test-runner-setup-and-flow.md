# Chatbot Test Runner Setup And Flow

## Purpose

This document explains the complete chatbot test runner setup currently implemented in `smart-messenger`.

It reflects the actual state of the feature after implementation:
- entity-created tables are used as the source of truth
- generic entity CRUD is kept in place
- custom code only handles orchestration and message-pipeline integration
- WhatsApp API calls are bypassed only for test messages

## Current Architecture

The chatbot test runner uses these parts:

1. Entity-created data tables
- `chatbot_test_cases`
- `chatbot_test_case_metas`
- `chatbot_test_runs`
- `chatbot_test_run_metas`
- `chatbot_test_run_items`
- `chatbot_test_run_item_metas`

2. Smart Messenger custom flow code
- `src/Services/ChatbotTestEntityService.php`
- `src/Jobs/MessageJobs/RunChatbotTestBatchJob.php`
- `src/Jobs/MessageJobs/SendWhatsAppReplyJob.php`
- `src/Http/Controllers/Api/ChatbotTestRunController.php`
- `src/Http/Controllers/ChatbotTestPageController.php`
- `resources/views/chatbot-tests/index.blade.php`
- `resources/views/chatbot-tests/show.blade.php`

3. Existing chatbot pipeline
- inbound `messages` row creation
- `ForwardToChatbotJob`
- chatbot-core outbound callback into `ChatbotV3OutboundJob`
- `ProcessChatbotResponseJob`
- `SendWhatsAppReplyJob`

## Important Schema Note

`chatbot_test_run_items` currently does **not** contain a direct `expected_contains` column.

Because of that:
- the run detail Blade view must not assume `$item->expected_contains` exists
- if needed later, `expected_contains` should either be added to the table or stored in meta and read safely

## Why A Dedicated Test Channel Is Required

The current message flow is channel-based.

The test runner creates real inbound `messages` rows, and downstream logic resolves context from `channel_id`.

That means a dedicated test channel is required for:
- organisation resolution
- chatbot integration resolution
- consistent message context

## How Start Run Works

When a run is started:

1. active chatbot test cases are loaded from `chatbot_test_cases`
2. one `chatbot_test_runs` row is created
3. one `chatbot_test_run_items` row is created per selected case
4. run meta `config` is stored in `chatbot_test_run_metas`
5. `RunChatbotTestBatchJob` is queued

## How RunChatbotTestBatchJob Works

Each execution of `RunChatbotTestBatchJob` does one step of the run.

1. load the run by `run_uid`
2. stop if run is `completed`, `failed`, or `cancelled`
3. finalize any previously `dispatched` items that now have outbound messages
4. find the next `pending` run item
5. if no pending item exists:
- update counters
- mark run `completed`
- store `summary` meta
6. if a pending item exists:
- create or update contact `9999999999 / Test Man`
- create a real inbound `messages` row
- mark inbound message meta:
  - `is_test = 1`
  - `chatbot_test_run_uid`
  - `chatbot_test_run_item_uid`
  - `chatbot_test_case_uid`
- update run item status to `dispatched`
- dispatch `ForwardToChatbotJob`
- queue the next `RunChatbotTestBatchJob` with delay

## How Test Messages Flow Through The Existing Pipeline

1. `RunChatbotTestBatchJob` creates inbound test message
2. `ForwardToChatbotJob` calls chatbot API
3. chatbot-core later inserts `ChatbotV3OutboundJob`
4. `ChatbotV3OutboundJob` dispatches `ProcessChatbotResponseJob`
5. `ProcessChatbotResponseJob` creates `SendWhatsAppReplyJob`
6. `SendWhatsAppReplyJob` detects `is_test = 1`
7. real WhatsApp API call is skipped
8. synthetic outbound `messages` row is saved locally
9. outbound message meta is linked back to the run item
10. next runner pass evaluates assertions and updates counters

## Test Mode Behavior In SendWhatsAppReplyJob

For test traffic only:
- no external WhatsApp API request is made
- a synthetic outbound message id is generated, such as `test-out-...`
- the outbound `messages` row is still inserted
- linkage meta is copied from inbound message to outbound message

This keeps test runs safe while preserving full message history.

## Queue Behavior In This Project

### BaseJob behavior
All jobs extend `BaseJob`.

`BaseJob` automatically sets queue name to the short class name.

Examples:
- `RunChatbotTestBatchJob`
- `ForwardToChatbotJob`
- `ChatbotV3OutboundJob`
- `SendWhatsAppReplyJob`

### Event-listener driven worker startup
The system starts workers through queue listeners.

When a job is queued:
- `JobQueuedListener` fires
- it calls `QueueManager->processQueues()`
- queue manager starts one worker for an active queue if allowed

### Important delayed job limitation
`RunChatbotTestBatchJob` currently re-queues itself with a delay.

That means:
- the next job is inserted immediately into `jobs`
- but its `available_at` is in the future
- the queue listener runs at insert time, not when delay expires
- therefore delayed jobs need periodic queue monitoring later

## Why A 5-Minute Delayed Job Does Not Continue By Itself

Example observed job row:

- `queue = RunChatbotTestBatchJob`
- `created_at = 1777180601`
- `available_at = 1777180901`

Difference:
- `300 seconds`
- which is `5 minutes`

So delay creation is correct.

But after 5 minutes:
- no new `JobQueuedListener` event fires
- no new worker is automatically started unless a periodic monitor runs

## Queue Registration Requirement

Because `QueueManager` only processes queues that exist in `queues` table, the queue must be registered.

Recommended queue rows:
- `RunChatbotTestBatchJob`
- `ForwardToChatbotJob`
- `ProcessChatbotResponseJob`
- `SendWhatsAppReplyJob`
- `ChatbotV3OutboundJob`

Example registration for the runner queue:


## Manual Worker Commands Used For Testing

### Runner queue only
```bash
php artisan queue:work database --queue=RunChatbotTestBatchJob --tries=3 --sleep=3 --timeout=120 --max-jobs=1 --stop-when-empty
```

### Outbound callback queue
```bash
php artisan queue:work database --queue=ChatbotV3OutboundJob --tries=3 --sleep=3 --timeout=120 --max-jobs=1 --stop-when-empty
```

### Notes
This project often uses short-lived workers.

That means each worker:
- processes one job
- exits

That model works for immediate jobs.

It does **not** automatically keep delayed jobs moving unless periodic queue monitoring is running.

## Recommended Operational Options

### Option 1. Always-on worker
Run a persistent queue worker.

Pros:
- simplest

Cons:
- not aligned with current listener-driven short-lived worker design

### Option 2. Periodic queue monitor
Run:
- `php artisan smart-messenger:monitor-queues`

through scheduler or recurring process.

Pros:
- aligns with your queue architecture
- delayed jobs can wake up when they become available

### Option 3. Best future design
Refactor the chatbot test runner to avoid delayed self-dispatch.

Instead:
- keep `next_dispatch_at` in `chatbot_test_runs`
- run a scheduled command every minute
- that command dispatches `RunChatbotTestBatchJob` only for due runs

This fits your short-lived worker model better than delayed chaining.

## Troubleshooting Notes

### 1. `Undefined property: stdClass::$expected_contains`
Cause:
- `chatbot_test_run_items` has no direct `expected_contains` field
- Blade view tried to access it directly

Fix:
- read safely from column or meta
- or add the field to run items if snapshotting is required

### 2. Run stays `pending`
Causes:
- no worker ran for `RunChatbotTestBatchJob`
- queue not registered in `queues`
- queue monitor not running

### 3. Run item stays `pending`
Causes:
- runner job did not execute
- first inbound test `messages` row was never created

### 4. Delayed next runner does not execute after 5 minutes
Cause:
- delayed job became available later
- no periodic queue monitor was running

### 5. Outbound test reply never saved
Causes:
- `ChatbotV3OutboundJob` never processed
- `ProcessChatbotResponseJob` or `SendWhatsAppReplyJob` queue not handled

## Recommended Next Improvement

The current implementation works, but for this codebase the best long-term design is:
- keep `next_dispatch_at` in run table
- create a scheduled command that finds due runs
- dispatch `RunChatbotTestBatchJob` immediately only when due
- avoid delayed self-dispatch inside the runner job

That would align better with the existing short-lived worker and queue-monitor architecture.
